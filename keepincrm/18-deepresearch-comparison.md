## РОЗБІР ПО КРИТЕРІЯМ

### 1. Rate Limits — КРИТИЧНА ВІДМІННІСТЬ

**Gemini:**
> «Жорсткий ліміт — до **5 запитів на секунду**. При перевищенні → 429 + `Retry-After`. Рекомендується Exponential Backoff.»

**ChatGPT:**
> «В офіційній документації **чітко не вказані** ліміти... рекомендовано не перевищувати кількасот запитів/хвилину... практично 10–20 req/sec.»

**Висновок:** Gemini дає конкретну цифру (5 req/sec). ChatGPT чесніший — визнає що не знає, але дає неточну оцінку (10–20 req/sec що суперечить реальності). Для інтегратора помилка у 4 рази (20 vs 5) — критична.

---

### 2. Структура кастомних полів — СУПЕРЕЧНІСТЬ

**Gemini (вірно):**
```json
"custom_fields": {
  "iedrpo_123": "01234567"
}
```
Передача як **об'єкт** з ID-ключами (унікальні системні ідентифікатори з префіксом).

**ChatGPT (можлива помилка):**
```json
"custom_fields": [
  {"name": "Адреса", "value": "Україна, Київ"}
]
```
Передача як **масив об'єктів** з текстовими назвами.

**Висновок:** Обидва формати можуть мати місце (старий vs новий API), але Gemini дає більш специфічний формат з реальними ID-ключами (`iedrpo_123`). ChatGPT-формат більш читабельний але може не спрацювати.

---

### 3. Retry-логіка вебхуків — КРИТИЧНА ВІДМІННІСТЬ

**Gemini:**
> «3 повторних спроби з інтервалом **15 хвилин** при відповіді 5xx»

**ChatGPT:**
> «Офіційно не задокументовано автоматичних повторних спроб. Реалізовувати на стороні клієнта.»

**Висновок:** Якщо Gemini правий — це важлива деталь для проектування надійних інтеграцій. ChatGPT не знає — і чесно каже про це.

---

### 4. Swagger / API-документація

**Gemini:**
- Посилання: `https://app.swaggerhub.com/apis-docs/KeepInCRM/keepincrm-api/`
- Згадує офіційну Postman-колекцію в кабінеті розробника

**ChatGPT:**
- «Swagger є на api.keepincrm.com але не завантажується статично»
- «Офіційної готової колекції Postman **немає**»

**Висновок:** Суперечність — один каже є Postman-колекція, інший каже немає. Gemini дає реальне посилання на SwaggerHub — це конкретно і перевіряємо.

---

### 5. Порядок міграції даних

**Gemini:**
```
1. Склад та Прайс-листи → отримати product_id
2. База Контрагентів    → отримати client_id
3. Угоди               → прив'язка до client_id + product_id
4. Завдання та Фінанси → прив'язка через parent_id
```
Починає зі **складу** — правильно, бо угоди мають товари які потребують product_id.

**ChatGPT:**
```
1. Клієнти/Контрагенти
2. Угоди
3. Завдання
```
Пропускає **Склад/Товари** на першому кроці — при міграції угод з товарами виникнуть помилки.

**Висновок:** Gemini правильніший — включає склад першим кроком.

---

### 6. Архітектурні патерни

**Gemini (унікально):**
- Event-driven архітектура з поясненням
- Loop Prevention для двосторонньої синхронізації: умова тригера `main_responsible_id_not_eq`
- Конкретна рекомендація: Redis Queue
- Тротлінг: мінімум 200 мс між запитами

**ChatGPT:**
- Загальний опис патернів без конкретних інструментів
- Рекомендує `external_id` для відстеження — корисна порада
- Тротлінг: 0.2–0.5 сек (аналогічно)

**Висновок:** Gemini глибший в архітектурних рекомендаціях.

---

### 7. Таблиця конекторів

**Gemini:**
- 19 інтеграцій з точними даними напрямку (двосторонній/односторонній)
- Деталізує що саме синхронізується по кожній
- Вказує Zakupivli.Pro та OpenAI (найновіші 2025)

**ChatGPT:**
- Більше інтеграцій у переліку (Ukr.net, Yahoo, Google Voice)
- Детальніший розбір деяких конекторів (LiqPay, Checkbox)
- Менш структурований

**Висновок:** Рівноцінно — у Gemini точніша структура, у ChatGPT ширший перелік.

### 8. Унікальні знахідки Gemini (відсутні у ChatGPT)

| Деталь | Значення |
|--------|----------|
| Rate limit = 5 req/sec | Критично для інтеграцій |
| Mustache-шаблонізатор для вихідних вебхуків | Знати синтаксис `{{#jobs}}...{{/jobs}}` |
| Файли — тільки через `comments` multipart/form-data | Типова пастка |
| Max file size = 100 Мб | Перевірка на стороні відправника |
| `total > 500.0` (decimal) в умовах тригерів | Тригер не спрацює з `500` без крапки |
| Аудит безпеки TAC Security (підтверджено Google) | Маркетинговий факт |
| Loop Prevention через умову тригера | Архітектурно важливо |
| 3 retry × 15 хв для вебхуків | Надійність системи |
| Postman-колекція в кабінеті розробника | Прискорює розробку |
| CORS — API тільки з Backend | Типова новачкова помилка |

---

### 9. Унікальні знахідки ChatGPT (відсутні у Gemini)

| Деталь | Значення |
|--------|----------|
| Email-інтеграції (Ukr.net, Yahoo, Zoho, Outlook) | Ширший перелік |
| Endpoint `/v1/agreements/:id/jobs` для товарів | Конкретний шлях |
| `assign_to_id` для завдань | Корисна деталь |
| Webhook.site для тестування | Практичний інструмент |
| HMAC Flask-приклад | Шаблон для власної верифікації |
| Чесність щодо відсутності Postman-колекції | Важливо не витрачати час |

## ПІДСУМКИ

### Де Gemini кращий
1. **Технічна точність** — конкретні цифри (5 req/sec, 15 хв retry, 100 Мб)
2. **Правильна структура даних** — ID-ключі кастомних полів (`iedrpo_123`)
3. **Правильний порядок міграції** — включає склад першим
4. **Архітектурні патерни** — Loop Prevention, Redis Queue, Event-driven
5. **Webhook деталі** — Mustache-шаблонізатор, retry, файли через multipart
6. **Тригери** — умова decimal (`500.0`), CORS заборонений з Frontend
7. **Swagger** — конкретне реальне посилання

### Де ChatGPT кращий
1. **Чесність** — відверто визнає що не знає точних лімітів
2. **Ширший список інтеграцій** — email-провайдери, Google Voice
3. **Webhook.site** — практичний інструмент для тестування
4. **HMAC-приклад** — навіть попри те що KeepinCRM не підтримує — як шаблон

### Ключовий висновок

**Gemini Deep Research для технічної документації KeepinCRM значно кращий.**

ChatGPT дав більш загальний звіт, місцями неточний (10–20 req/sec замість 5, пропущений склад в міграції), але чесніший у визнанні невизначеності.

**Для бази знань:** файл від Gemini (16-technical-guide.md) є основним технічним довідником. Файл від ChatGPT (17-technical-guide-chatgpt.md) — допоміжний, корисний для cross-reference та ширшого переліку email-інтеграцій.
