# KeyCRM — REST API (2025–2026)

> Джерело: KeyCRM технічний довідник та API, 2025–2026

## Базова інформація
- **Base URL:** `https://openapi.keycrm.app/v1`
- **Auth:** `Authorization: Bearer <token>` — тільки на backend, CORS блокує браузер
- **Rate limit:** 60 запитів/хвилину — пауза 1 сек між запитами
- **Пагінація:** max 50 записів на сторінку (default 15)
- **Формат:** JSON з пагінацією (`data` + `meta`)
- **OpenAPI spec:** `https://keycrm.s3.de.io.cloud.ovh.net/static/open-api.yml`

## Отримання API-ключа
Налаштування → Основні → рядок «API-ключ» → «Згенерувати API-ключ»

## Приклад авторизації
```bash
curl -X GET "https://openapi.keycrm.app/v1/order" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Accept: application/json"
```

## CRUD по сутностях

### Замовлення `/v1/order`
| Метод | Endpoint | Опис |
|-------|----------|------|
| GET | `/v1/order` | Список з фільтрами (`status_id`, `created_between`, `manager_id`) |
| POST | `/v1/order` | Створення (обов'язково: `source_id` + `buyer`) |
| PUT | `/v1/order/{id}` | Оновлення статусу, коментарів, кастомних полів, тегів |
| DELETE | — | ❌ Заблоковано (фінансова цілісність) |

### Контакти/Покупці `/v1/buyer`
| Метод | Опис |
|-------|------|
| GET | Список покупців |
| POST | Створення (обов'язково: `full_name` АБО `phone`/`email`) |
| PUT | Оновлення контактів, кастомних полів, адрес доставки |
| DELETE | ⚠️ Тільки Admin/Owner |

### Товари `/v1/products`
| Метод | Опис |
|-------|------|
| POST | Створення (обов'язково: `name`, `sku`, `price`) |
| DELETE | ❌ Не рекомендується — використовувати деактивацію |

### Pipeline-картки `/v1/pipelines/cards`
| Метод | Опис |
|-------|------|
| POST | Створення (обов'язково: `contact` з `full_name`/`phone`/`email`) |
| PUT | Оновлення стадії, суми, менеджерів, кастомних полів |
| DELETE | ❌ Заблоковано |

> ⚠️ **Задачі (Tasks):** окремого endpoint `/tasks` в REST API v1 **немає**. Обхід — системні коментарі + призначення `assigned_id`.

## Фільтрація / Пагінація / Сортування
```
# Фільтрація
GET /v1/order?filter[status_id]=5&filter[created_between]=2026-05-01,2026-05-26

# Пагінація (default 15, max 50)
GET /v1/order?page=2&limit=50

# Сортування (дефіс = DESC)
GET /v1/order?sort=-id
```

## Batch Import
- `/v1/buyer/import` — до 100 об'єктів/запит
- `/v1/order/import` — до 100 об'єктів/запит

## Ролі та права доступу
- API-ключ успадковує права ролі користувача, який його згенерував
- Редагування товарів → потрібна роль «Edit Products»
- Редагування покупців → потрібна «Edit Buyers and Companies»
- Видалення → тільки Owner/Admin

## Коди помилок
| Код | Причина | Дія |
|-----|---------|-----|
| 400 | Синтаксична помилка JSON | Перевірити структуру тіла запиту |
| 401 | Невірний/відсутній токен | Перевірити префікс `Bearer ` (з пробілом!) |
| 403 | Немає прав у ролі | Перевірити права користувача API-ключа |
| 404 | Сутність видалена або не існує | Обробити gracefully |
| 422 | Помилка валідації | Перевірити формат (особливо кастомні поля) |
| 429 | Перевищено ліміт 60 req/min | Читати `Retry-After` + exponential backoff з jitter |
| 500 | Внутрішня помилка | Retry з exponential backoff |

## Топ помилок інтеграторів
| Симптом | Причина | Вирішення |
|---------|---------|-----------|
| 401 при валідному токені | Відсутній пробіл: `Bearer<token>` | `Bearer <token>` (з пробілом) |
| 422 на кастомних полях | Рядок замість масиву | `["значення"]` а не `"значення"` |
| 429 у циклах | Немає затримок між запитами | Пауза 1 сек між запитами |
| CORS помилка | API-виклики з браузерного JS | Тільки через backend |
| Дублі контактів | Різний формат телефону | Стандартизувати до `+380XXXXXXXXX` |
| Збій SKU-синхронізації | Регістр/пробіл в артикулі | SKU ідентичні на обох платформах |
| 403 на редагування товару | Немає ролі «Edit Products» | Перевірити права в налаштуваннях |

## Паттерни інтеграції
1. **One-Way:** лендинг → `POST /v1/pipelines/cards`
2. **Two-Way:** захист від петель через session flag / request ID
3. **Batch Import:** `/v1/buyer/import`, `/v1/order/import`
4. **Event-Driven:** webhook → негайний 200 → Redis/RabbitMQ → async workers

## Безпека
- Сервери в Амстердамі та Фінляндії
- Всі дані шифруються при передачі
- API-токен ніколи не розкривати на клієнтській стороні
- CORS блокує прямі запити з браузера
