# Odoo MRP (Manufacturing) — відео-конспекти YouTube

> Джерело: офіційні YouTube-відео Odoo (транскрипти), 2026-06

---

## Огляд MRP (MRP Overview)

- Odoo MRP охоплює повний цикл виробництва: інженерія → BoM → закупівлі → виробництво → контроль якості → обслуговування → доставка
- Єдина база даних для всіх модулів: планувальники бачать реальні залишки запасів у реальному часі
- Основні модулі, що взаємодіють з MRP: Inventory, Purchase, Shop Floor, Quality, PLM, Maintenance, Barcode
- Процес: дефіцит компонентів → авто-RFQ → прийом на склад → резервування компонентів → запуск виробництва
- Підтримується субпідряд для аутсорсингу специфічних або надлишкових замовлень
- Shop Floor: оператори реєструють процес одним натиском (старт/пауза/підтвердження) — прогрес-бар МО оновлюється миттєво
- Після доставки: обробка повернень → QCP → ECO через PLM (з контролем версій) → превентивне ТО через Maintenance

## Специфікації матеріалів (Bill of Materials — BoM)

### Основи BoM
- BoM (Bill of Materials) — «рецепт» виробу: список компонентів з кількостями + операції (робочі замовлення)
- Активація Work Orders: Manufacturing → Configuration → Settings → увімкнути "Work Orders"
- Два способи створення BoM:
  1. Manufacturing → Products → Bill of Materials → New
  2. З картки продукту → смарт-кнопка "Bill of Materials" → New
- На один продукт можна створити кілька BoM; перший у списку — стандартний для нових МО
- Поле **Reference** — код для розрізнення BoM одного продукту
- Поле **BoM Type**: `Manufacture this product` або `Kit`
- Поле **Quantity** — кількість одиниць, що виготовляється за цим BoM (зазвичай 1)

### Вкладка Components
- Список компонентів із потрібними кількостями
- Кількості автоматично масштабуються при зміні кількості в МО

### Вкладка Operations
- Операція = робоче замовлення (Work Order)
- Поля операції: назва, робочий центр, тривалість
- **Cost based on**: `Actual time` (реальні трудовитрати для кожного МО) або `Theoretical time` (однакова тривалість для всіх МО)
- **Duration computation**: `Computed` (ODoo розраховує на основі N останніх WO) або `Fixed` (фіксована тривалість)
- **Instructions** — інструкції, таблиці, файли/діаграми для операції

### Вкладка Miscellaneous
- **Manufacturing Lead Time** — скільки днів займає виробництво
- **Days to Prepare MO** — скільки днів потрібно для збору компонентів перед виробництвом; можна розрахувати автоматично кнопкою "Compute" (береться максимальний термін поставки серед всіх компонентів)
- **Flexible Consumption**: `Allowed`, `Allowed with warning`, `Blocked`
- **Operation Dependencies** — увімкнення залежностей між WO (з'являється після активації відповідного налаштування)

## Виробничі замовлення (Manufacturing Orders — MO)

### Створення МО
- Manufacturing → Operations → Manufacturing Orders → New
- МО прив'язане до одного продукту та однієї кількості
- При виборі продукту: автоматично підставляється BoM, компоненти, WO
- При виборі BoM до продукту: поле продукту заповнюється автоматично (зворотний порядок)
- Зміна кількості → автоматичне оновлення кількостей компонентів і очікуваної тривалості WO

### Статуси МО
- **Ready** (зелений) — всі компоненти доступні (1-step manufacturing)
- **Ready** (помаранчевий) — компоненти доступні, але ще не переміщені до виробничого місця (2/3-step)
- **Waiting Another Operation** — потрібно спочатку завершити трансфер компонентів
- **Not Available** — компонентів не вистачає

### Поля МО
- **Scheduled Date / Scheduled End** — дати плану; End = Start + сума тривалостей WO (не редагується вручну)
- **Component Status** — зелений (доступно) / помаранчевий (очікує переміщення) / "Not Available" (дефіцит)
- **Work Orders tab** — список WO з колонками: робочий центр, очікувана тривалість, реальна тривалість, статус
- **Miscellaneous tab** — source document (для MTO/реорд. правил відображає джерело замовлення)

### Обробка МО
- WO статуси: Ready → In Progress (натиск Start) → Finished (натиск Done)
- Наступний WO стає Ready тільки після завершення попереднього (якщо є залежності)
- Завершення МО: кнопка **Produce All** → МО закривається, склад оновлюється
- Альтернативно: через Shop Floor (Start → Mark as Done → Close Production)

## Робочі центри (Work Centers)

### Огляд
- Work Center — місце виконання виробничих операцій (верстат, дільниця, стенд)
- Dashboard MRP: картки WC з кількістю готових WO та поточним OEE
- Кнопка монітора на картці → перехід у Shop Floor

### Налаштування WC
Manufacturing → Configuration → Work Centers → New

| Поле | Опис |
|------|------|
| Name / Code / Tag | Ідентифікатори |
| Working Hours | Розклад (стандарт: 40 год/тиж, Пн-Пт 8:00-17:00) |
| Alternative Work Center | Альтернативний WC для автоматичного перенаправлення |
| Time Efficiency | % ефективності; 100% = WO виконується рівно за очікуваний час; 200% = вдвічі швидше; 50% = вдвічі повільніше |
| Capacity | Кількість одиниць, що обробляються одночасно |
| OEE Target | Цільовий % продуктивності (стандарт 90%) |
| Setup Time | Час підготовки перед WO (додається до кожного WO) |
| Cleanup Time | Час прибирання після WO (додається до кожного WO) |
| Allowed Employees | Обмеження доступу (якщо порожнє — будь-хто може працювати) |

### Вкладка Product Capacities
- Специфічна потужність для окремих продуктів (замість загальної)
- Дозволяє задати різні capacity та setup/cleanup для різних виробів

### Розрахунок тривалості WO
Формула: `(Базовий час × кількість ÷ capacity ÷ (time_efficiency/100)) + setup + cleanup`

**Приклад**: 10 лавок × 10 хв/шт = 100 хв; Time Efficiency 50% → 200 хв; Capacity 5 → 40 хв; Setup 15 + Cleanup 10 = **65 хв**

## Залежності робочих операцій (Work Order Dependencies)

- Активація: Manufacturing → Configuration → Settings → Work Orders + Work Order Dependencies
- Налаштування у BoM → вкладка Miscellaneous → увімкнути "Operation Dependencies"
- Після цього у деталях операції (вкладка Operations) з'являється поле **Blocked By**
- Приклад: "Attach Legs to Table Top" заблоковано → "Cut Table Legs"
- Відображення залежностей на вкладці Operations: кнопка "Extra Options" → показати колонку Blocked By
- У МО: заблокований WO має статус "Waiting for another WO"
- У планувальнику (Planning by Production): залежний WO відображається після батьківського; стрілка між ними
- Перетягування залежного WO раніше батьківського → блок виділяється червоним

## Кроки виробництва (1/2/3 Steps)

### Налаштування
- Inventory → Configuration → Settings → увімкнути **Multi-Step Routes**
- Inventory → Configuration → Warehouses → поле **Manufacture** → вибір кількості кроків
- Налаштування на рівні складу: різні склади можуть мати різну кількість кроків

### 1-Step (за замовчуванням)
- Лише МО; трансфери компонентів і готової продукції не відстежуються
- Запаси оновлюються автоматично (без валідації трансферів)
- МО статус → Ready, як тільки компоненти є в наявності
- Підходить для малих виробництв

### 2-Step (Pick Components → Manufacture)
- Одна додаткова операція: переміщення компонентів на виробничу дільницю
- МО статус = "Waiting Another Operation" до валідації трансферу
- Компонент-статус помаранчевий (Available але очікує переміщення)
- Після валідації трансферу → МО стає Ready (зелений)

### 3-Step (Pick → Manufacture → Store)
- Ще одна операція: переміщення готової продукції на склад
- Після виробництва → SFP-трансфер (Store Finished Product) потрібно валідувати
- Беклоги можливі на кожному з трьох кроків

## Беклоги виробництва (Manufacturing Backorders)

### Що таке беклог
- Виробничий беклог — неповне МО, решту якого буде оброблено пізніше
- Запуск: при реєстрації виробленої кількості, меншої за планову

### Через Shop Floor
- Натиснути "Register Production" → ввести фактичну кількість → Validate → Mark As Done
- МО автоматично розділяється: перший МО (фактичний) + беклог МО (залишок)
- Нумерація: оригінальний МО → МО-001 (вироблено) + МО-002 (беклог)

### При нестачі компонентів (3-step)
- При валідації Pick-трансферу з неповним набором компонентів → Odoo пропонує створити беклог
- Кожен крок (Pick, Manufacture, Store) може мати свій беклог
- Залишок потрібних компонентів відображається у Inventory → Replenishment для поповнення
- Після надходження нових компонентів і поповнення → відповідний Pick-трансфер стає Ready

## Планування виробництва (Manufacturing Planning)

### Кнопка Plan
- Доступна після підтвердження МО → "Plan" → Odoo резервує WC відповідно до його розкладу та потужності
- Масове планування: відмітити кілька МО → кнопка Plan (батч-операція)

### Перегляд планування
- **Planning → Planning by Production**: план за МО (список або Gantt)
- **Planning → Planning by Work Center**: план за WC (Gantt, тижневий вигляд)

### Логіка планування
- Кожна операція планується на найранішій доступний час після попередньої операції (у межах того ж МО)
- Якщо основний WC зайнятий → Odoo автоматично перевіряє альтернативний WC
- Блоки одного кольору на Gantt = одне МО

### Індикатори конфліктів
- Фіолетова іконка ока = WO очікує на завершення іншого WO
- Червоний знак оклику = WO заплановано одночасно з іншим WO
- Кнопка **Replan** — автоматичне вирішення конфліктів планування

## Терміни виробництва (Manufacturing Lead Times)

### Типи lead times
| Тип | Де задається | Опис |
|-----|-------------|------|
| Customer Lead Time | Картка продукту → Inventory tab → Logistics | Днів від підтвердження замовлення до відвантаження |
| Manufacturing Lead Time | BoM → Miscellaneous tab | Днів для виробництва продукту |
| Days to Prepare MO | BoM → Miscellaneous tab | Днів для збору компонентів/субзбірок перед виробництвом |
| Vendor Delivery Lead Time | Компонент → Purchase tab → Vendor | Днів від замовлення до отримання компонента |

- Customer Lead Time — незалежна від інших, задається вручну
- Days to Prepare MO можна розрахувати автоматично кнопкою "Compute" (= max терміну поставки серед усіх компонентів)
- Прогноз доступності: МО-форма → кнопка "View Forecast" → Manufacturing Forecast → BoM Overview

## Ефективність виробництва (OEE — Overall Equipment Effectiveness)

### Концепція OEE
- OEE — % «дійсно продуктивного» часу роботи WC
- Odoo розраховує OEE автоматично на основі дій операторів
- Цільовий OEE задається на картці WC (за замовчуванням 90%)

### Статуси часу WC
| Статус | Умова |
|--------|-------|
| **Fully Productive** | WO завершено в межах очікуваної тривалості |
| **Reduced Speed** | WO перевищив очікувану тривалість (надлишковий час — reduced speed) |
| **Blocked** | WC заблоковано (вказується причина: Equipment Failure, Material Availability, Process Defect, Setup, тощо) |

### Перегляд OEE
- Dashboard WC: кнопка OEE на картці → деталі по даному WC
- Reporting → Overall Equipment Effectiveness → агрегований звіт по всіх WC
- Угруповання за замовчуванням: Work Center + Loss Reason
- Ручне додавання причини втрати: у звіті → New (Loss Reason, дата)
- Блокування/розблокування WC також доступне з Shop Floor (три крапки → Block/Unblock)

## Make-to-Order (MTO) в MRP

### Що таке MTO
- MTO (Make-to-Order) = «Replenish on Order» — виробництво або закупівля ініціюється тільки після підтвердження замовлення
- MTO маршрут заархівований за замовчуванням (не всі компанії його використовують)

### Активація MTO
1. Inventory → Configuration → Warehouses → Routes → Filters: Archived → обрати MTO → Actions → Unarchive

### Налаштування продукту для MTO
- Картка продукту → Inventory tab → Operations → Routes:
  - `Replenish on Order (MTO)` — обов'язково
  - `Manufacture` або `Buy` — обов'язково (MTO не працює самостійно)
- Якщо `Manufacture` → при підтвердженні SO автоматично створюється МО
- Якщо `Buy` → при підтвердженні SO автоматично створюється PO

### Особливості MTO
- МО завжди прив'язується до SO (Source document видно у Miscellaneous tab МО)
- Продукт резервується тільки для конкретного SO (не може бути використаний для іншого)
- МО створюється навіть якщо продукт є на складі (MTO ігнорує наявний запас)

### Альтернатива: Reordering Rule з Min=0, Max=0
- Той самий ефект: МО створюється автоматично при підтвердженні SO
- Відмінність: продукт **не резервується** для конкретного SO (може бути використаний для інших)
- Source document у МО: "OP" (Order Point)
- Налаштування: Inventory → Configuration → Reordering Rules → New → Route: Manufacture, Min=0, Max=0, To Order=1

## З'єднання Sales Order → Manufacturing Order

### Метод 1: MTO маршрут
- SO підтверджено → автоматично з'являється смарт-кнопка Manufacturing
- МО прив'язано до SO; продукт зарезервовано для цього SO

### Метод 2: Reordering Rule (0/0)
- SO підтверджено → МО генерується, але не прив'язується до SO
- Продукт може використовуватись для будь-якого SO

## Під-збірки (Sub-Assemblies)

### Концепція
- Субзбірка — компонент, що сам виробляється перед використанням у фінальному продукті
- Приклад: «Wheelie Leg» (підзбірка) → використовується у «Wheelie Table» (фінальний виріб)

### Налаштування
- Компонент-підзбірка: картка продукту → Inventory tab → Routes → `Replenish on Order (MTO)` + `Manufacture`
- BoM підзбірки: створити окремий BoM зі своїми компонентами та операціями

### Поведінка системи
- При підтвердженні батьківського МО → автоматично створюється дочірнє МО для підзбірки (Child MO)
- Кнопка **Child MO** з'являється на батьківському МО
- Дочірнє МО містить смарт-кнопку Source MO (посилання на батьківське)
- Прогноз: компонент-підзбірка в батьківському МО показує дату завершення дочірнього МО

### Планування субзбірок
- Масове планування: виділити обидва МО → Plan
- Дочірнє МО планується першим; батьківське — одразу після завершення дочірнього
- Глибина вкладеності: необмежена (A → B → C → D... )

## Набори (Product Kits)

### Що таке Kit
- Kit — набір компонентів, що продаються або виробляються разом
- BoM Type: `Kit`

### Особливості Kit
- Кількість в наявності Kit = кількість наборів, що можна зібрати з наявних компонентів
- Kit **не потрібно фізично збирати** перед продажем; при відвантаженні він розкладається на компоненти
- При додаванні Kit до SO: у замовленні покупця Kit відображається як один рядок; у Delivery Order — розгортається в окремі компоненти
- Kit можна додавати як компонент до інших BoM (зручно для стандартизації)
- При створенні МО: компоненти Kit автоматично підставляються (Kit сам не з'являється в МО)

## Побічні продукти (By-Products)

### Активація
- Manufacturing → Configuration → Settings → увімкнути **By-Products**

### Налаштування у BoM
- Вкладка **By-Products** → Add a line
- Поля: продукт, кількість, **Produced in Operation** (на якому WO утворюється побічний продукт)

### Поведінка
- У МО: вкладка By-Products з чекбоксом "Produced" (відмічається після завершення відповідного WO)
- Після завершення МО: залишки побічного продукту зараховуються на склад
- Побічні продукти можна використовувати як компоненти для інших МО

## Гнучке споживання (Flexible Consumption)

### Режими (налаштовуються у BoM → Miscellaneous tab)
| Режим | Поведінка |
|-------|-----------|
| **Allowed** | Вільне споживання будь-якої кількості без обмежень |
| **Allowed with Warning** | Дозволено, але при закритті МО з'являється попередження для підтвердження |
| **Blocked** | Без прав адміна МО не можна закрити при розбіжності (показується кнопка "Continue Production") |

### Індикатори споживання у вкладці Components
- Помаранчевий — перевитрата (consumed > expected)
- Чорний — недовитрата (consumed < expected)
- Зелений — точно за BoM

### Особливості
- Зміна спожитих кількостей можлива безпосередньо у МО (без Tablet View)
- Фактичні кількості впливають на вартість виробництва
- Незаповнені залишки (unconsumed) — показуються окремо
- Підтримується також для субпідрядного виробництва

## Брак та розбирання (Scrap & Unbuild)

### Scrap Orders (Брак)

**Доступ до браку**:
- Зі сторінки МО: кнопка **Scrap** у верхній панелі

**1-step виробництво — Scrap компонента**:
- До виготовлення фінального виробу: можна списати лише компоненти (не фінальний виріб)
- Після виготовлення: з'являється можливість списати готовий виріб
- Чекбокс **Replenish Quantities** → автоматично створює PO або МО для заміни

**2/3-step — Scrap після Pick**:
- Якщо є запас — береться з поточного запасу; PO/МО не створюється
- Якщо запасу немає — створюється новий Pick-трансфер для поповнення

**Смарт-кнопки після браку**: Scraps (всі замовлення браку по МО) + Purchases (замовлення на поповнення)

### Unbuild Orders (Розбирання)

**Доступ**: Manufacturing → Operations → Unbuild Orders → New

**Поля**:
- Product, BoM (підставляється автоматично), Quantity
- Manufacturing Order (за яким виготовлено виріб)
- Source Location (звідки забрати), Destination Location (куди повернути компоненти)

**Результат**: кількість виробу зменшується, кількість компонентів збільшується відповідно до BoM

**Розбирання виробу з нульовим запасом**: Odoo попереджає про можливі невідповідності в обліку, але дозволяє продовжити

## Розділення та об'єднання МО (Split and Merge)

### Split (Розділення)
- МО → шестерня (gear icon) → **Split**
- Поля у вікні Split: кількість нових МО, дата, відповідальний, кількість для кожного
- Сума кількостей у нових МО = кількість оригінального МО (перевірка системи)
- Нові МО: той самий номер + суфікс 001, 002, 003...
- Перегляд усіх частин: смарт-кнопка **Back Orders**

### Merge (Об'єднання)
- Умова: усі МО мають бути для того самого продукту
- Manufacturing Orders → відмітити кілька МО → Actions → **Merge**
- Об'єднаному МО присвоюється перший доступний номер
- Дата = найраніша із дат вихідних МО
- Номери вихідних МО відображаються у колонці Source об'єднаного МО

## Витрати виробництва (Production Costs)

### Налаштування
1. **Payroll**: задати hourly rate в картці співробітника (Settings tab) — оновлюється вручну
2. **Product Category**: Costing Method = AVCO або FIFO (для коректної оцінки; Standard Price не перераховує автоматично)
3. **Work Center**: поле "Cost per Hour" (вартість роботи верстату/дільниці на годину)
4. **BoM Operations**: тривалість операцій визначає суму трудових витрат

### Структура витрат МО (Cost Analysis tab)
- **Component Costs**: вартість кожного компонента (на основі stock valuation)
- **Operation Costs**: вартість роботи WC (Cost per Hour × тривалість)
- **Labor Costs**: зарплата оператора (Hourly Rate × реальна тривалість WO)
- **Unit Cost**: загальна вартість виробництва одиниці

### Перегляд витрат
- Завершений МО → **Overview** → Cost Analysis (потрібен Developer Mode для кнопки Valuation)
- **Valuation** смарт-кнопка → які компоненти списані та за якою ціною
- Картка продукту → **Compute Price from BoM** — перерахунок середньої вартості на основі реальних МО
- **Cost Analysis** смарт-кнопка → загальна картина по всіх МО для продукту

### Трейл вартості (Developer Mode)
МО → Valuation → компонент → Purchase → Receipt → Valuation Layer (звідки береться ціна)

## Основний виробничий план (Master Production Schedule — MPS)

### Активація та налаштування
- Manufacturing → Configuration → Settings → увімкнути **Master Production Schedule**
- Додаткові параметри: часовий горизонт перегляду та кількість колонок

### Додавання продуктів до MPS
- Planning → Master Production Schedule → **Add a Product**
- Вибір продукту та BoM
- Параметри: Safety Stock Target, Min/Max репленішменту

### Рядки MPS (за замовчуванням)
| Рядок | Опис |
|-------|------|
| Forecasted Demand | Прогнозований попит (вводиться вручну) |
| Suggested Replenishment | Рекомендована кількість для поповнення (авто) |
| Forecasted Stock | Прогнозований залишок |

### Додаткові рядки (кнопка "Rows")
- **Actual Demand** — підтверджені SO (дріб від Forecasted Demand)
- **Actual Demand Y-1** — попит за аналогічний період минулого року
- **Actual Replenishment** — підтверджені РО/МО (дріб від Suggested Replenishment)
- **Available to Promise** — скільки можна пообіцяти клієнтам (= Replenishment − Actual Demand)

### Кольори клітинок (Actual Replenishment)
- **Сірий** — фактичне = запропоноване
- **Червоний** — поповнили більше ніж потрібно
- **Жовтий/помаранчевий** — поповнили менше ніж потрібно → від'ємний прогнозований залишок

### Дія кнопки Replenish
- Натискання "Replenish" для конкретного місяця → створює МО або RFQ (залежно від маршруту компонента)
- Для компонентів що купуються → RFQ
- Для компонентів що виробляються → МО
- Перегляд документів: натиснути на число в рядку Actual Replenishment

## Методи поповнення в MRP (Replenishment Methods)

### Reordering Rules для компонентів
- Налаштування у Inventory → Configuration → Reordering Rules або на картці продукту → смарт-кнопка
- Поля: Min Quantity, Max Quantity, To Order (кількість замовлення)
- **Важливо**: у Purchase tab продукту потрібно вказати vendor (Odoo замовляє у першого у списку)
- При падінні запасу нижче Min → авто-RFQ на (Max − поточний запас + потреба МО)

**Приклад**: Min=10, Max=100, потреба МО=3 → замовляємо 103 або 101 (залежно від запасу)

### MTO маршрут для виробників
- Продукт → Inventory tab → Routes → `Replenish on Order (MTO)` + `Manufacture`
- Детально описано у розділі "Make-to-Order (MTO)"

### Reordering Rule з нульовими Min/Max (альтернатива MTO)
- Min=0, Max=0, To Order=1, Route=Manufacture
- При підтвердженні SO → МО генерується автоматично (без прив'язки до SO)
- Source document у МО: "OP" (Order Point)
- Корисно також для керування субзбірками у багаторівневих BoM

## Engineer-to-Order (ETO)

### Відмінність від MTO
- **MTO**: дизайн вже готовий, виробляємо тільки при замовленні
- **ETO**: дизайн ще не існує, ми його розробляємо разом з клієнтом

### Налаштування для ETO
1. Accounting → Settings → увімкнути **Analytic Accounting**
2. Employees: задати **Timesheet Cost** (вартість/год) у Settings tab кожного співробітника
3. Project → Settings → увімкнути **Timesheets**
4. Sales → Settings → Invoicing → `Invoice what is delivered`
5. Створити **Project Template** (напр. "ETO Design Service") з типовими задачами (Design, Review)
6. Створити **Service Product** (ETO Design Service):
   - Product Type: Service
   - Create on Order: Project (з вибором шаблону)
   - Ціна = вартість/год
7. Створити **Generic ETO Product**:
   - Product Type: Storable
   - Inventory Routes: MTO + Manufacture
   - BoM Template (порожній, для дублювання)

### ETO Workflow
1. Створити SO: додати ETO Service (оцінка годин) + ETO Product (ціна = $1 тимчасово)
2. Підтвердження SO → автоматично: аналітичний рахунок + проект + завдання + МО
3. Дизайнер логує час у Timesheets задачі
4. Інженер логує час після рев'ю
5. Оновити BoM: дублювати шаблон, додати компоненти та операції, прив'язати до аналітичного рахунку
6. У МО: змінити BoM на конкретний (замість шаблону) → підтвердити → обробити
7. Оновити SO: фактична кількість годин + ціна готового виробу
8. Виставити рахунок → Confirm → Register Payment
9. Доставка → Validate

### Звітність ETO
- Project app → три крапки → Dashboard → Revenues, Costs, Profit

## Субпідряд (Subcontracting)

### Активація
- Manufacturing → Configuration → Settings → увімкнути **Subcontracting**
- Purchase → Settings → увімкнути **Dropshipping** (для схем з дропшипінгом)

### Базовий субпідряд (Basic Subcontracting)
- Субпідрядник сам забезпечує компоненти
- BoM: тип `Subcontracting`, без компонентів; вказати субпідрядника
- Продукт → Purchase tab → субпідрядник як Vendor
- Маршрут продукту: `Buy` (або Dropship)
- Процес: PO → підтвердження → субпідрядник виготовляє → доставка → Receipt → Validate

### Субпідряд з постачанням компонентів (Resupply Subcontractor on Order)
- Ми постачаємо компоненти субпідряднику зі свого складу
- BoM: тип `Subcontracting`, **вказати компоненти**
- Кожен компонент → Inventory tab → Routes → `Resupply Subcontractor on Order`
- Після підтвердження PO: з'являється смарт-кнопка **Resupply** (трансфер компонентів на адресу субпідрядника)
- Процес: PO → Resupply (відправити компоненти) → субпідрядник виготовляє → Receipt

### Субпідряд з дропшипінгом компонентів (Dropship Subcontractor on Order)
- Компоненти купуємо у постачальника і відправляємо напряму субпідряднику
- Компонент → Purchase tab → Vendor (хто поставляє субпідряднику)
- Компонент → Inventory tab → Routes → `Dropship Subcontractor on Order`
- Після підтвердження PO (субпідрядника): генерується RFQ для постачальника компонентів
- Після підтвердження RFQ (постачальника): з'являється Dropship-документ → Validate
- Після цього на PO субпідрядника з'являється Resupply смарт-кнопка

### Повний дропшипінг (Subcontracting + Dropshipping)
- Компоненти йдуть від постачальника до субпідрядника
- Готовий виріб іде від субпідрядника до клієнта (минаючи нас)
- Маршрут продукту: `Dropship`
- Продукт → Purchase tab → Subcontracted: увімкнути
- Lead times: Vendor Delivery (дні до субпідрядника) + Manufacturing (дні виробництва) + доставка клієнту = Customer Lead Time
- Після підтвердження SO → авто-RFQ для субпідрядника

## Обслуговування обладнання (Maintenance)

### Maintenance Requests — створення
Чотири способи:
1. Maintenance app → Maintenance → Maintenance Requests → New
2. Maintenance app → Maintenance → Maintenance Calendar → клік на клітинку → Create
3. Shop Floor → WO → gear icon → Request Maintenance
4. Manufacturing → МО → кнопка **Maintenance Request**

### Поля форми запиту
| Поле | Опис |
|------|------|
| Name | Назва запиту |
| For | Equipment або Work Center |
| Maintenance Type | Corrective (виправлення) або Preventative (профілактика) |
| Manufacturing Order / Work Order | Прив'язка до МО/WO (якщо є) |
| Team / Responsible | Команда та відповідальний |
| Scheduled Date / Duration | Дата та тривалість |
| Priority | 1-3 зірки |
| Block Work Center | Блокує WC під час обслуговування |

### Управління запитами
- Kanban-вигляд: Maintenance → Maintenance Requests
- Налаштування стадій: gear icon на стадії → Edit → "Request Confirmed" → блокує WC на вказану тривалість при переміщенні запиту в цю стадію
- Maintenance Calendar: перетягування для перепланування

### Обладнання (Equipment)
- Maintenance → Equipment → Machines and Tools
- Поля Equipment: категорія, відповідальний, WC/Location, серійний номер, гарантія
- **Maintenance tab** Equipment:
  - Preventive Maintenance Frequency (частота профілактики)
  - Maintenance Duration (тривалість)
  - Expected MTBF (очікуваний час між відмовами)
  - Real MTBF (розрахований Odoo на основі реальних запитів)
  - Estimated Next Failure (прогноз наступної відмови)
  - MTTR (Mean Time to Repair — середній час ремонту)
- Авто-генерація профілактичних запитів: 1 раз на добу при досягненні порогу

### Рекурентне обслуговування
- У формі Maintenance Request: Maintenance Type = Preventative → з'являється чекбокс **Recurrent**
- Параметри рекурентності: repeat every [N] [days/weeks/months/years], до [дата] або forever
- Наступний запит у серії відображається в Kanban після закриття поточного
- Видалення будь-якого запиту в серії → видалення всієї серії
- Зняття чекбоксу Recurrent → видаляє всі майбутні, залишає лише найближчий

## Штрихкоди в MRP (Consuming Components Using Barcodes)

### Активація
- Settings → Inventory → Barcode Scanner → увімкнути
- Штрихкоди продуктів мають бути налаштовані (Settings → Configure Barcodes)

### Способи відкриття МО у Barcode app
- Сканування QR-коду/штрихкоду роздрукованого МО

### Швидке споживання всіх компонентів
- Сканувати штрихкод **фінального виробу** → всі компоненти підставляються → кнопка **Produce All**

### Споживання невідстежуваних компонентів (3 способи)
1. Значок олівця → ручне введення кількості
2. Кнопка "+1" (або "+N" для всієї кількості) → авто-споживання
3. Сканування штрихкоду компонента → авто-реєстрація

### Споживання відстежуваних компонентів (з лотами/SN)
- Значок олівця → вибір лоту, введення кількості → Confirm
- Сканування штрихкоду **лоту** (рекомендовано, а не штрихкоду продукту)

### Брак через Barcode
- Gear icon → Scrap → вибір продукту та лоту, введення кількості → Scrap → Confirm

### Додавання непланових компонентів
- Кнопка **Add Component** → сканування штрихкоду нового компонента → з'являється в МО

### Завершення
- Сканування штрихкоду фінального виробу або кнопка "+1" → **Produce All** → підтвердження (може з'явитись Consumption Warning)
