# Odoo Studio — Knowledge Base (Video Transcripts)

## Що таке Studio

- Odoo Studio — застосунок для кастомізації бази даних без знань програмування
- Дозволяє: додавати поля, створювати нові views, редагувати документи (інвойси, звіти), автоматизувати процеси, створювати нові застосунки з нуля
- **Коли варто використовувати Studio**: коли немає стандартного функціоналу і кастомізація збереже достатньо часу. Розрахунок: (час на задачу зараз − час з кастомізацією) × кількість разів за 5 років > час на впровадження
- **Коли не варто**: якщо впровадження займе кілька днів, а економія — лічені години за роки
- **Не намагайтеся відтворити старі системи** — типова помилка при впровадженні
- Усі стандартні застосунки Odoo (60+, 1500+ моделей) покривають більшість типових бізнес-сценаріїв — спочатку шукайте там
- Кастомізації зроблені в Studio зберігаються в окремому модулі й **автоматично переносяться при оновленні** версії Odoo (на відміну від прямої правки views)
- Studio не призначений для зміни бізнес-логіки — для цього потрібна розробка. Studio — для полів, невеликих автоматизацій, кастомних views
- При роботі зі Studio спочатку тестуйте зміни на **дублікаті бази даних**

## Основні можливості

- Додавання та редагування полів у будь-якій моделі
- Управління views (form, list, kanban, search, activity, graph)
- Створення та редагування PDF-звітів
- Налаштування approval rules (правил погодження)
- Створення automation rules (правил автоматизації)
- Додавання меню та підменю до застосунків
- Налаштування фільтрів та доменів
- Додавання smart buttons на форми
- Створення нових застосунків з нуля
- **Studio 19 (нові можливості)**: AI-поля, AI у automation rules, нові тригери on create / on create and edit, динамічні followers в автоматизаціях, HTML editor у automation rules, sequence у automation rules, chatter в automation rules для колаборації, column width у list view, зручний Ctrl+X / Ctrl+V у PDF-редакторі

## Поля: прості та реляційні

### Прості поля (Simple Fields)

- **Text** — текстове поле
- **Multi-line text** — багаторядковий текст
- **Selection** — вибір з кількох фіксованих значень (widget: dropdown або radio)
- **Checkbox** — булеве поле (так/ні)
- **Date / DateTime** — дата та час
- **Monetary** — грошове значення
- **Decimal** — числове поле з десятковою частиною

**Властивості полів** (налаштовуються через ліву панель у Studio):
- Invisible / Required / Read-only — можна задати безумовно або через **умову (conditional)**
- Label — назва поля
- Help tooltip — підказка для користувача
- Default value
- Widget — спосіб відображення (наприклад, radio замість dropdown для selection-поля з небагатьма варіантами)

**Приклад**: поле "contact type" (owner / seeker / both) з widget=radio і horizontal-відображенням на формі контактів

**Зміни зберігаються автоматично** при переході між полями в Studio

### Реляційні поля (Relational Fields)

**Many-to-One (M2O)** — багато записів посилаються на один. Приклад: багато sales orders → один контакт (customer). Основна перевага — уникнення дублювання даних (адреса, email зберігаються один раз)

**One-to-Many (O2M)** — один запис містить посилання на багато. Приклад: у контакті відображаються всі пов'язані sales order lines. **Для O2M обов'язково повинен існувати M2O на протилежному боці**

**Many-to-Many (M2M)** — багато записів пов'язані з багатьма. Типовий приклад: теги (tags). Продукт може мати багато тегів, тег — пов'язаний з багатьма продуктами

**Related field** — не створює нове відношення, а відображає поле з пов'язаної моделі через існуючий зв'язок. Приклад: показати email контакта на формі sales order (через наявний M2O customer → contact)

**Додавання полів у Studio**:
1. Перевірте існуючі поля (може вже існувати) через пошук у списку Existing Fields
2. Якщо не знайшли — створіть нове
3. Нові поля можна drag-and-drop у потрібне місце форми
4. Поле, створене в одному view, можна повторно використати в інших view через "existing fields"

**Поле можна зробити required** — тоді при збереженні без заповнення буде помилка

## Фільтри та домени

### Меню та підменю
- У Studio можна додавати нові пункти меню та підменю для моделей застосунку
- Підменю створюється через слайдання пунктів вправо у списку меню
- Меню можна drag-and-drop для зміни порядку

### Filter Rules (Правила фільтрів)
- Прив'язуються до пунктів меню
- Можна зробити фільтр **default** — він автоматично застосовується при відкритті меню
- Приклад: підменю "Seekers" має filter rule де contact_type = seeker OR both

### Домени (Domains)
Домени використовуються у кількох місцях:

1. **Filter rules** — фільтрація записів при відкритті меню
2. **Властивості полів** — обмеження вибору в relational полях. Приклад: у полі "owner" на формі нерухомості домен обмежує вибір тільки контактами з типом owner або both
3. **Search view filters** — кастомні фільтри в search panel (наприклад, "For rent" / "For sale")
4. **Automation rules** — умова для спрацювання правила (наприклад, value >= 1,000,000)

**Синтаксис домену**: умови через AND/OR, можна змінити через UI (перемикач any/all)

### Smart Buttons
- Додаються на форму для швидкого доступу до пов'язаних записів
- Для додавання кнопки між моделями повинен існувати relational зв'язок
- Приклад: кнопка "Properties" на формі контакта — показує всю нерухомість цього власника

## Правила погодження (Approval Rules)

- Approval rules прив'язуються до **кнопок** на формах
- Налаштовуються в Studio: клік на кнопку → вкладка Properties → Approvals → Add approval step
- При додаванні кроку погодження на кнопці з'являється аватар

**Що можна налаштувати**:
- Хто повинен погодити (конкретний користувач або роль)
- Кого сповістити після погодження
- Умова спрацювання (domain): наприклад, total > 1,000 EUR
- Кілька кроків погодження (можна задати порядок)

**Як працює**:
1. Користувач натискає кнопку (наприклад, Pay) — якщо approval не наданий, з'являється попередження
2. Для призначеного approver створюється activity
3. Approver бачить запис у списку з pending approval
4. Approver відкриває запис, перевіряє деталі, натискає Approve або платить напряму
5. У chatter проставляється мітка "approved", призначений користувач отримує сповіщення

**Приклад**: vendor bills > 1,000 EUR потребують погодження CEO перед оплатою

## Автоматизації

### Загальне
- Automation rules — автоматичні дії при настанні певної умови
- Налаштовуються через Studio → Automations або напряму з kanban-view (іконка ⚙ поруч із назвою стадії → Automations)
- При відкритті через Studio фільтрація за поточною моделлю вже застосована

### Тригери (Triggers)

| Тригер | Коли спрацьовує |
|--------|----------------|
| On Save | При збереженні запису (треба вказати поле-тригер) |
| On Create | При створенні нового запису |
| On Create and Edit | При створенні або редагуванні |
| Stage is set to | При переміщенні запису в певну стадію (kanban) |
| Tag is added | При додаванні тегу |

**Studio 19**: тригери "On Create" та "On Create and Edit" додані для спрощення (раніше треба було хитрі умови через "ID is not set")

### Дії (Actions)

- **Update Record** — оновити поле(я) запису. З Studio 19: можна оновити з допомогою AI (Update with AI), додати Sequence, оновити HTML-поле через HTML editor
- **Create Activity** — створити активність (to-do, call тощо). Можна використовувати dynamic user (наприклад, агент з поля на записі) замість конкретного користувача. Для dynamic user потрібно вказати технічне ім'я поля (видно в developer mode або в Studio tooltip)
- **Add Followers** — додати фоловерів до запису. З Studio 19: підтримка **dynamic followers** (наприклад, parent company контакта)
- **Send Email** — надіслати email за шаблоном

### Умови (Domain у автоматизації)
- Automation спрацьовує лише якщо запис відповідає заданому домену
- Приклад: value >= 1,000,000 — тільки для нерухомості вартістю від мільйона

### Studio 19 — нові можливості автоматизацій
- **Sequence** у автоматизації: оновити поле значенням sequence (раніше потребувало Python-коду). Приклад: автоматична назва проекту у форматі PROD/2025/001/MOVIE
- **Dynamic followers**: `add followers → type: dynamic → field: parent_id` — додає материнську компанію як фоловера
- **HTML editor** в Update Record: можна оновлювати HTML-поля (description) з форматуванням через slash commands
- **Chatter в automation rules**: для колаборації між командою при налаштуванні автоматизацій
- **Чейнування автоматизацій**: розбивайте складні автоматизації на кілька простих правил з чіткими промптами для AI (LLM краще виконує одне завдання)

### Приклад повної автоматизації (VIP properties)
```
Тригер: On Save (поле: value)
Умова: value >= 1,000,000
Дії:
  1. Update Record: agent = поточний користувач
  2. Update Record: owner.tags → Add tag "VIP"
```

## Звіти: створення та кастомізація

### Кастомізація layout всіх звітів
- Settings → Company → Configure Document Layout
- Тут: логотип компанії, кольори, тема (folder та ін.)
- Зміни застосовуються до всіх документів

### Редагування існуючих звітів
1. Відкрийте застосунок → Studio → Reports
2. Звіти відфільтровані за поточною моделлю (можна прибрати фільтр)
3. **Перед редагуванням — зробіть дублікат** (три крапки → Duplicate), вимкніть оригінал з print menu, перейменуйте
4. **Синій фон** = placeholder (буде замінено реальними даними при генерації)
5. **Пунктирна рамка** = умовний блок (відображається залежно від умов)

**Що можна редагувати**:
- Видаляти/додавати колонки в dynamic table
- Змінювати форматування (колір, розмір шрифту) — наприклад, виділити підсумкову суму
- Додавати поля через slash + field selector
- Для розширеного редагування — Edit Sources (XML)

**Studio 19 — PDF reports**:
- Зручний field selector з показом шляху до поля
- Переміщення полів через Ctrl+X / Ctrl+V (раніше треба було видалити і додати знову)

### Створення нового звіту з нуля

1. Studio → Reports → New
2. Виберіть тип: **Blank** (без header/footer), External, Internal
3. Додавайте контент через slash commands:
   - `/field` — поле з моделі
   - `/bullet list` — маркований список
   - `/header` — заголовок
   - `/table` — таблиця (вкажіть кількість рядків/колонок)
   - `/banner info` — інформаційний банер
4. Для посилань на поля пов'язаних моделей — використовуйте `/field` → вкажіть шлях через стрілки

**Приклад структури звіту для нерухомості**:
- H1: назва об'єкта (field: display_name)
- Bullet list: адреса, дата доступності, email та телефон агента
- Текстове поле: опис об'єкта
- Image field: фото об'єкта
- H2: "General Information"
- Table 8×2: тип, ціна, площа, кімнати, ванні, площа саду, гараж

## Кастомізація інтерфейсу (Views)

### Типи views
- **Form** — перегляд та редагування одного запису
- **List** — огляд списку записів
- **Kanban** — управління потоком (стадії)
- **Search** — налаштування фільтрів, групувань, пошуку
- **Activity** — огляд запланованих активностей
- **Graph** — візуалізація даних для звітності

### Управління views через Studio
- Перемикання між views: через іконки або Views tab
- **Додавання/видалення полів**: у list/form view — drag-and-drop або кнопка видалення. У kanban/graph — лише existing fields
- **Нові поля** можна створювати лише у form та list view
- **Make invisible** — поле залишається в базі, але не відображається. Для перегляду invisible-полів: Views tab → Show invisible elements (striped background)
- **Set as Default view**: Views → три крапки → Set as default (змінює view за замовчуванням при відкритті)
- **Disable/Enable views**: через три крапки в Views tab

### View-специфічні налаштування
- **List view**: can create / can edit / can delete (наприклад, заборонити видалення в конкретному view)
- **Kanban view**: інші опції порівняно з list/form
- **Search view**: group by поля, autocomplete поля для пошуку, кастомні фільтри з доменами

### Приклади кастомізації
- Додати поле "for rent / for sale" на kanban-картку → зробити bold
- Додати group by "Property Type" до search view
- Додати адресу до autocomplete в пошуку
- Видалити поле "Salesperson" з list view (або зробити invisible)
- Активувати Activity view для моделі

### Column width (Studio 19)
- У list view для кожного поля можна задати ширину колонки (раніше лише через XML)

## Кастомізація додатків

### Додавання полів до моделі
Рекомендований підхід:
1. Відкрийте form view моделі → Studio
2. Додайте всі потрібні поля у form view (вони потім доступні для інших views через "existing fields")
3. Для relational полів: визначте тип відношення (M2O, O2M, M2M), цільову модель
4. Для related полів: вкажіть шлях через relational поле (наприклад, owner → email)

**Типові поля для кастомного застосунку нерухомості**:
- Selection: "for rent / for sale"
- Many-to-One: owner → Contact (з required=True)
- Related: owner.email, owner.phone (з відповідним widget)
- Selection: property type (house, apartment, land, offices)
- Multi-line text: address
- Числові: bedrooms, bathrooms, surface, garden_space
- Boolean: garage
- Monetary: value
- Image: picture

### Кастомізація Search View
- Group by: додайте поля для групування (type, rent_or_sale, responsible)
- Autocomplete: додайте address для пошуку за адресою
- Filters: кастомні фільтри з доменами (for rent / for sale)

## Створення нових додатків

### Кроки створення застосунку
1. Studio → New App icon
2. Вкажіть назву застосунку та назву першої моделі
3. Виберіть features для моделі:
   - **Chatter** (default) — коментарі та активності
   - **Custom sorting** (default) — ручне сортування записів
   - **Archiving** (default) — архівування записів
   - **Pipeline stages** — kanban-стадії
   - **User assignment** — відповідальний користувач
   - **Picture** — поле зображення
   - **Monetary value** — грошове поле

### Автоматично створюється при новій моделі
- Два Access Control Lists:
  - Administrators: всі права
  - Internal users: всі права **крім** видалення записів
- Кілька views за замовчуванням (kanban, list, form тощо)
- Пункт меню у домашньому меню

### Коли створювати новий застосунок vs використовувати існуючий
- **Існуючий**: якщо бізнес-концепція вже представлена (контакти, продажі, нерухомість як проект)
- **Новий**: якщо потрібна специфічна модель з унікальними полями та процесами, яка не вкладається в жоден стандартний застосунок
- **Не reinvent the wheel** — завжди спочатку перевіряйте 60+ стандартних застосунків Odoo

### Доступ до нового застосунку
- Після закриття Studio → новий застосунок з'являється в Home menu
- Автоматично доступні: kanban view, picture field, responsible, value field, chatter

## Що нового в Studio (v19)

### AI + Studio
- **AI-поля в Studio**: новий тип поля "AI field" — drag and drop → вкажіть тип (text, multi-line) → напишіть prompt → додайте field selector (`/` → field selector)
- Приклад AI-поля: аналіз chatter і відображення "actor satisfaction score 1-5" без читання всіх повідомлень
- **AI в automation rules**: тип дії "Update with AI" → вкажіть поле для оновлення → напишіть prompt з field selector та record selector
- Приклад: автоматично призначати відповідального менеджера продажів на основі preferred_genres з AI-поля на контакті
- **AI не потребує коштів за промпти** (при SaaS) але потрібні API-ключі OpenAI/Gemini для sh/on-premise

### Нові можливості автоматизацій (детально)
- **Sequence**: нова дія в Update Record — оновити поле значенням sequence без Python-коду
- **Dynamic followers**: при "Add followers" тепер є тип "dynamic" — вибираєте поле-джерело (наприклад, parent company контакта)
- **HTML editor**: у Update Record можна редагувати HTML-поля через знайомий редактор зі slash-командами
- **Нові тригери**: On Create, On Create and Edit — замінюють складні умови через "ID is not set"
- **Chatter в automation rules**: для обговорення та tracking змін у правилах без виходу з Studio

### Нові можливості у fields, views, PDF

**Дати (v19)**:
- Якщо рік поточний — рік не відображається (лише місяць та день)
- Якщо рік інший — рік відображається
- Можна змінити формат дати через Studios field properties

**PDF reports (v19)**:
- Зручний field selector показує повний шлях до поля при доданні
- Ctrl+X / Ctrl+V для переміщення елементів у звіті (замість видалення та повторного додавання)

**List view (v19)**:
- Column width для будь-якого поля через Studio без редагування XML

### Рекомендації щодо Studio vs розробки
- **Studio**: поля, невеликі автоматизації, views, звіти, approval rules
- **Розробка**: зміна бізнес-логіки, складні інтеграції через API, webhook
- **Кнопки в Studio**: можна створювати кнопки, що викликають server actions або методи; можна робити smart buttons для відображення даних з інших моделей
