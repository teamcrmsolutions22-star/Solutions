# Pipedrive — Модель даних та Архітектура

## Структура зв'язків

- **Deal ↔ Person:** Угода обов'язково прив'язана до одного основного контакту; через API можна додавати Participants для складних B2B-взаємодій
- **Deal ↔ Organization:** Угода може існувати без організації (B2C), але якщо є — зв'язок автоматично успадковується через Person
- **Lead vs Deal:** Lead — спрощена структура, окремі API-ендпоінти `/leads`; успадковує метадані та кастомні поля Deal; при конвертації всі дані переносяться в Deal автоматично

## Кастомні поля в API (технічно)

При створенні кастомного поля Pipedrive присвоює **унікальний 40-символьний хеш-код** (наприклад, `dcf558aac1ae4e8c4f849ba5e668430d8df9be12`).

**Взаємодія через API — тільки через хеш-код**, не через Label/назву.

### Приклад: створення угоди з кастомним полем

```http
POST https://api.pipedrive.com/v1/deals?api_token=YOUR_API_TOKEN
Content-Type: application/json

{
  "title": "Угода з великим дистриб'ютором",
  "value": 15000,
  "currency": "USD",
  "dcf558aac1ae4e8c4f849ba5e668430d8df9be12": "Значення кастомного текстового поля"
}
```

### Отримання хеш-кодів полів

| Сутність | Ендпоінт |
|----------|----------|
| Угоди | `GET /v1/dealFields` |
| Організації | `GET /v1/organizationFields` |
| Контакти | `GET /v1/personFields` |
| Товари | `GET /v1/productFields` |

У відповіді кастомні поля ідентифікуються: `"edit_flag": true`

### Особливості форматів кастомних полів

**Monetary (грошовий тип):**
- Основний ключ (наприклад, `ffk9s9`) — числове значення
- Додатковий ключ `ffk9s9_currency` — ISO-код валюти (наприклад, `"USD"`)

**Single-option / Multiple-option:**
- При оновленні через API передавати **числові id** опцій, не текстові значення

## Нотатки про невизначеності

- Чи можна обійти обов'язкові поля через API — не задокументовано ⚠️
- Soft delete vs hard delete — не задокументовано ⚠️
- Формат часових зон для timestamps — не задокументовано ⚠️
- Робота з товарами в угоді (`Deal Products`) — окремий sub-endpoint, не через основний PUT ⚠️
