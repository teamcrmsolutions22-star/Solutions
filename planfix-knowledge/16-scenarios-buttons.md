# Planfix — Сценарії та Кнопки (детально)

> Джерело: YouTube-канал Planfix (транскрипти ~50 відео)

---

## Структура сценарію (3 розділи / 6 блоків)

**Розділ 1:** Подія-тригер + умови  
**Розділ 2:** Додаткові умови / логіка фільтру  
**Розділ 3:** Дії + затримка і повторення

1. **Подія** — тригер
2. **Умови** — фільтр відповідності
3. **Черга і терміни** — затримка та перевірки
4. **Від чийого імені** — автор дії
5. **Операції** — дії
6. **Сповіщення** — повідомляти / не повідомляти

## Ліміти сценаріїв

| Обмеження | Значення |
|-----------|---------|
| Масова дія сценарієм | 150 задач |
| Кнопка: натискань | 1 раз/хвилину на одну задачу |
| Логіка фільтру | Максимум 250 символів |
| Цикл (сценарій сам себе) | Блокується після 2-ї ітерації |
| HTTP-запит | >15 000 мс → червоний у лозі |
| Мінімум між повтореннями | 5 хвилин |

## Тригери (події запуску)

### Зміна полів
- `Змінено значення поля` — будь-яке системне або кастомне поле
- Доступні попередні значення для порівняння

### Створення та відповідність умовам
- `Задача створена і відповідає умовам`
- `Задача створена або змінена і стала відповідати умовам` — два в одному
- `Задача змінена і стала відповідати умовам`
- `Задача змінена але все ще відповідає умовам`
- `Задача прийнята і відповідає умовам`

> ⚠️ Перехід із Чернетки в будь-який статус = «задача створена»

### Видалення та об'єднання
- `Задача видалена і відповідає умовам` — можна автоматично відновити
- `Задача об'єднана з іншою задачею` — незворотний процес

### Дати
- `Настала дата задачі і задача відповідає умовам`
- Підтримує обчислення: «через 2 дні після дати», «за 2 дні до дати»
- Опції: `з урахуванням робочого часу` / `без урахування`

### Коментарі
- `Додано коментар і задача відповідає умовам`
- `Прочитано повідомлення на пошту і задача відповідає умовам` — друга галочка в звіті відправки
- `Сталася помилка відправки` — червоний перекреслений гурток (пошта, месенджери)

### Аналітики
- `Додано будь-яку аналітику і задача відповідає умовам`
- `Додано один рядок аналітик` — з аналізом даних конкретного рядка
- `Змінено один рядок аналітик` — для перерахунку обчислюваних полів

### Чек-листи
- `Додано пункт листа і задача відповідає умовам`
- `Змінено пункт чек-листу`

## Умови (фільтр)

### Оператори
| Оператор | Опис |
|----------|------|
| `дорівнює` | Точний збіг |
| `не дорівнює` | Будь-яке інше |
| `більше або дорівнює` | ≥ |
| `менше або дорівнює` | ≤ |
| `містить` | Пошук по тексту |
| `містить взагалі <поле>` | Перевірка наявності значення |
| `не містить взагалі` | Поле порожнє |

### Параметри фільтру
- **Система:** автор зміни, день тижня, подія викликана іншим сценарієм
- **Задача:** статус, виконавець, аудитори, постановник, проект, дата, над/підзадачі
- **Контакти:** групи, компанії, кастомні поля в ролях
- **Коментарі:** текст, джерело, наявність файлу, тип аналітики, хто повідомлений

> ⚠️ Ліміт 250 символів на всю логіку фільтру (включаючи І, АБО, дужки)

### Логіка: `І` / `АБО` з дужками для групування

## Змінні (%%% синтаксис)

```
%%%Сьогодні%%%          # поточна дата
%%%День тижня%%%        # день тижня
%%%Номер задачі%%%
%%%Ім'я контакту%%%
%%%<ПовнаНазваПоля>%%%  # будь-яке поле задачі
```

> ⚠️ Змінні НЕ спрацьовують в задачах зі статусом «Чернетка»

## IF/ELSE в операціях

```
ЯКЩО <умова>
  [дія 1]
ІНАКШЕ
  [дія 2]
```

**Підводний камінь:** якщо умова `ЯКЩО` не виконана і блок `ІНАКШЕ` відсутній → в задачу приходить порожній коментар.

## Дії (операції)

### Поточна задача
- `Змінити цю задачу` — поля, статус, виконавців, аудиторів, дати, назву, опис
- `Додати коментар` — з %%%змінними%%%, скрити від виконавця, вказати отримувача
- `Відправити листа` — шаблон, отримувач, відправник

### Інші задачі
- `Змінити іншу задачу` — лише першу знайдену ⚠️
- `Змінити інші задачі` — всі знайдені ✅
- `Створити задачу` — за шаблоном, як підзадачу або окрему, з визначенням проекту

### Контакти та аналітики
- `Змінити контакт`
- `Додати аналітику` — вибір типу, заповнення полів з задачі або фіксованих значень
- `Створити задачі з аналітики`

### Комунікація
- `Відправити SMS`
- `Написати в загальний чат`
- `Відправити HTTP запит` — відправка в API, отримання і розбір відповідей

### Спеціальні
- `Натиснути на кнопку` — викликати дію кнопки автоматично
- `Запустити користувацьку операцію сценарію`
- `Відновити задачу` — якщо була видалена

## Черга і терміни виконання

| Варіант | Поведінка |
|---------|-----------|
| Негайно | Спрацює одразу при події |
| Через вказаний час | Мінімум 5 хвилин для хвилинного інтервалу |
| З урахуванням робочого часу | Якщо подія в неробочий час → перший робочий день |

### Додаткові перевірки при відкладанні
- `Без перевірок` — спрацює через час
- `Якщо продовжує відповідати умовам` — перевірить умови заново
- `Якщо виконує інші умови` — задати окремі умови для перевірки

**Опція:** `Нове спрацьовування перезапускає відлік` — кожна нова подія перезапускає таймер

## Від чийого імені

- `Автор зміни` (за замовч.)
- `Постановник задачі`
- `Конкретний співробітник` (часто «Робот» — приховує автоматизацію від звичайних співробітників)

---

## Сповіщення

- **Повідомляти** (за замовч.) — сповіщення відправляються
- **Не повідомляти** — зміни від сценарію приховані
- ⚠️ Виняток: якщо дія `Додати коментар` вказує конкретну роль/співробітника — сповіщення **завжди** відправляється

## 7 паттернів інтегратора

**1. Завдання-метроном**
Службова задача + сценарій рухає дату кожні N годин → джерело точного часу. Для запуску сценарію в конкретний час.

**2. Команда в полі**
Текстове поле + значення = команда → другий сценарій реагує. Для розгалуженої логіки.

**3. Службове завдання-агент**
КОС з API-запитом пошуку → результат у поле «Службова задача» → «Змінити іншу задачу» → видалити. Для зміни задачі знайденої за умовою.

**4. Тексти в довіднику**
Тексти шаблонів у довіднику → сценарій читає → при зміні тексту правимо тільки довідник.

**5. Лог через аналітику**
Поле змінилось → рядок аналітики: «назва поля», «значення», «хто змінив», «дата».

**6. GIF-індикатор завантаження**
Довідник з GIF → обчислюване поле показує GIF доки чекаємо відповідь API.

**7. Тема як маршрутизатор**
Зміна поля «Тема» → типові дії. Параметри в довіднику.

## SLA-події (листопад 2025)

- `Перевищено час очікування клієнтом відповіді` — таймер від останнього повідомлення клієнта

## Кнопки

### Відмінність від сценаріїв
- Замість тригера → **умови відображення**
- Хто може натиснути: за роллю, конкретний співробітник
- Виконуються вручну (але сценарій може натиснути кнопку автоматично)

### Обмеження ⚠️
- Доступні лише на **Plan B / Plan X**
- Хвилинний антиспам-обмежувач на одну кнопку
- Кнопка має бути в тому ж процесі/об'єкті для спрацьовування

### Структура кнопки
1. Назва і опис
2. Умови відображення
3. Права доступу (хто може натиснути)
4. Дії при натисканні (ті ж самі, що в сценаріях)

## Користувацькі операції сценаріїв

- Багаторазове використання однотипної дії в різних сценаріях/кнопках
- Розташування: `Управління акаунтом → Системні довідники → Користувацька операція сценарію`
- Заповнення: змінні з задачі, контакту, обчислювані інфоблоки, значення аналітик

## Порядок діагностики

| Крок | Інструмент | Де знайти |
|------|-----------|-----------|
| 1 | Технічний лог задачі | `Задача → Лог → Технічний лог` |
| 2 | Тест сценарію на задачі | `Сценарій → ▶ → Тестувати на задачі` (вставити URL) |
| 3 | Лог сценаріїв | `Управління акаунтом → Процеси → <Об'єкт> → Сценарії → Лог` |

### Типові причини несрацьовування

| Причина | Рішення |
|---------|---------|
| Процес задачі ≠ процесу сценарію | Перевірити процес в сценарії і задачі |
| Умови не виконуються | Тест сценарію на задачі |
| Подія вже відбулась до додавання сценарію | Сценарії діють тільки на нові події |
| Задача в статусі «Чернетка» | Перейти в активний статус |
| Черга сценаріїв конфліктує | Змінити порядок сценаріїв в події |
