## 4 типи email-адрес

### 1. Внутрішні адреси (Внутренние адреса)
Генеруються автоматично, налаштування не потрібне.

**Email проекту:**
- Вхідні листи → задачі в проекті
- Відправник листа = постановник задачі
- Формат: `project@account.planfix.info`

**Email співробітника:**
- Вхідні листи → задачі співробітника
- Співробітник = виконавець задачі
- Формат: `employee@account.planfix.info`

### 2. Віртуальна адреса (Виртуальный адрес)
- Єдина адреса для всього акаунту
- Приймає листи від клієнтів
- При відповіді в задачі → вибір адреси відправника:
  - Особиста/адреса співробітника
  - Корпоративна віртуальна адреса
- Налаштовується в налаштуваннях акаунту

### 3. Зовнішні адреси (Внешние адреса)
- Для відправки вихідних листів
- Налаштування обов'язкове
- Потрібно налаштувати для компанії або співробітника
- Протокол: **POP-3** має бути увімкнений

**Параметри конфігурації:**
- Поштовий сервер
- Ім'я відправника
- Адреса електронної пошти
- Співробітники, які можуть використовувати

### 4. (Системні) Email для обробки правил
- Адреси, що тригерять правила обробки пошти
- Налаштовуються у «Правилах обробки пошти»

---

## Правила обробки пошти (Правила обработки почты)

**Розташування:** `Налаштування акаунту → Задачі по пошті`

### Структура правила
1. **Назва правила**
2. **Умови спрацьовування** — текст листа містить певні слова
3. **Блок логіки фільтру** — `І` / `АБО`
4. **Додаткові операції** — дії при спрацьовуванні

### Умови
- `Вміст листа містить текст` — часткові слова (без закінчень) для охоплення словоформ
- Кілька умов з `І` або `АБО`

### Дії (операції) при спрацьовуванні
- `Додати виконавця`
- `Змінити статус` — наприклад, «в роботі»
- Інші зміни задачі

**Приклад правила:**
```
Тригер: лист містить «баг» АБО «помилка» АБО «не працює»
Дія: Додати «Служба підтримки» як виконавця
      Змінити статус → «в роботі»
```

## Пріоритет підпису (4 рівні)

Підпис у вихідних листах визначається за пріоритетом (від вищого до нижчого):

1. **Підпис шаблону листа** — якщо шаблон містить підпис
2. **Підпис акаунту** — загальний корпоративний підпис
3. **Підпис проекту** — підпис, налаштований для проекту
4. **Підпис співробітника** — особистий підпис

## Трекінг email

### Статус відправки
- Дві галочки в звіті відправки:
  - Перша = лист відправлено
  - Друга = **прочитано** (через піксель трекінгу)

### Журнал вхідних листів
- Зберігання: **15 днів** ⚠️
- Після 15 днів — видаляється

### Подія сценарію для трекінгу
- `Прочитано повідомлення на пошту і задача відповідає умовам` — тригер для автоматизації після прочитання
- `Сталася помилка відправки` — червоний перекреслений гурток (пошта, месенджери)

## Email-автоматизація

### Обробка вхідної пошти в планувальнику
- Вхідна пошта обробляється через «Роботу з поштою» в розкладі
- Правила застосовуються автоматично при надходженні листів

### Відповідь клієнту з задачі
- Коментар у задачі → автоматично відправляється клієнту
- Вибір адреси: особиста або корпоративна
- Шаблони листів з %%%змінними%%%

### Масове розсилання
- Заблоковане для груп контактів як ролей задач (захист від спаму)
- Для масових розсилань — окремі інструменти

## Підводні камені email

| Ситуація | Наслідок |
|---------|---------|
| Email контакту в спамі | Блокуються всі сповіщення до відповіді |
| Протокол POP-3 вимкнено | Зовнішня адреса не працює |
| Правило без блоку `ІНАКШЕ` | Пустий коментар у задачі |
| Трекінг-піксель заблокований | Другої галочки не буде |
| Журнал вхідних листів | Зберігається лише 15 днів |
